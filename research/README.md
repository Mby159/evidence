# Evidence 项目研究报告

## 一、项目定义

**Evidence** — 利用区块链技术固定电子证据（视频、照片、音频、文字），实现防篡改、可追溯、可验证的证据保全系统。

核心流程：文件 → 哈希 → 存证 → 验证

### 设计原则

1. **C端优先** — 面向个人用户，解决"拍了照片想固定时间戳"的痛点
2. **模块化** — 每一层可替换（哈希/存储/链/签名），方便独立迭代
3. **MVP 先行** — 第一版只做哈希 + 本地存储 + 验证，不碰链
4. **Web3 存储** — 利用 IPFS/Filecoin 等分布式存储，零成本、去重、隐私

### 模块架构

```
┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐
│  Hash    │→ │  Store   │→ │  Chain   │→ │  Verify  │
│ SHA-256  │  │ Local    │  │ (可选)   │  │ 比对哈希  │
│ SHA-3    │  │ IPFS     │  │ 公链 L2  │  │          │
│ BLAKE3   │  │ Arweave  │  │ 本地链   │  │          │
└──────────┘  └──────────┘  └──────────┘  └──────────┘
```

## 二、法律框架（中国）

### 2.1 关键法规

| 时间 | 法规 | 要点 |
|------|------|------|
| 2018.09 | 最高法《关于互联网法院审理案件若干问题的规定》 | **首次明确**区块链存证的法律效力，第11条：当事人提交的电子数据，通过区块链等技术手段进行存证固定的，人民法院应当予以确认 |
| 2021.06 | 最高法《人民法院在线诉讼规则》 | 第16-18条：区块链存证的审查标准，强调"上链前"和"上链后"数据完整性 |
| 2020 | 《最高人民法院关于民事诉讼证据的若干规定》修订 | 完善电子数据的认定规则 |

### 2.2 法律效力的关键要素

法院审查区块链证据时关注：

1. **上链前数据真实性** — 生成时是否被篡改（来源可信）
2. **上链过程可靠性** — 存证平台是否中立、技术是否可靠
3. **上链后数据完整性** — 哈希是否一致
4. **身份关联** — 谁存的、能否证明是本人操作

### 2.3 已有判例

- **杭州互联网法院（2018）**：全国首例区块链存证案，确认区块链证据效力
- **北京互联网法院**：多起案件采纳区块链证据
- **广州互联网法院**：使用"网通法链"系统

| 产品 | 特点 |

### 3.3 现有产品的共同问题

- **成本高**：按量收费，大规模存证成本不可控
- **中心化依赖**：大部分依赖平台运营，平台倒闭证据可能丢失
- **隐私担忧**：文件上传到第三方服务器
- **门槛高**：面向企业，个人用户难用

## 四、技术架构

### 4.1 核心流程

```
原始文件 → 提取元数据 → 计算哈希(SHA-256) → 数字签名 → 写入区块链
                                                        ↓
验证时：重新计算哈希 → 与链上哈希比对 → 验证数字签名 → 输出结果
```

### 4.2 哈希算法选择

| 算法 | 安全性 | 性能 | 推荐 |
|------|--------|------|------|
| SHA-256 | 高 | 快 | **首选**，行业标准 |
| SHA-3 | 高 | 略慢 | 备选，抗碰撞性更强 |
| BLAKE2 | 高 | 极快 | 性能优先场景 |
| BLAKE3 | 高 | 更快 | 新一代，但生态较新 |

### 4.3 区块链选型

#### 公链方案

| 链 | 优势 | 劣势 | 成本 |
|------|------|------|------|
| **Bitcoin (OP_RETURN)** | 最安全、最去中心化 | 仅存 40 字节（刚好一个哈希），速度慢 | ~$0.5-2/笔 |
| **Ethereum L1** | 生态丰富、智能合约灵活 | Gas 费高 | ~$0.5-10/笔 |
| **Ethereum L2 (Arbitrum/Base)** | 便宜、继承 L1 安全性 | 依赖 L1 | ~$0.01-0.1/笔 |
| **Solana** | 速度快、成本低 | 历史上有过宕机 | ~$0.001/笔 |

#### 国内联盟链

| 链 | 运营方 | 特点 |
|------|--------|------|
| **长安链 (ChainMaker)** | 北京微芯研究院 | 国产开源，政府支持 |
| **FISCO BCOS** | 微众银行 | 开源，金融级 |
| **蚂蚁链** | 蚂蚁集团 | 企业级，需付费 |
| **BSN** | 中国移动 | 区块链服务网络 |

#### 本地链

| 模式 | 适用场景 | 风险 |
|------|----------|------|
| 纯本地链 | 个人/内部审计 | 第三方不信、单点故障 |
| **本地链 + 公链锚定** | **推荐方案** | 需定期锚定 |
| 多节点私有链 | 小团队/公司 | 节点数少，安全性有限 |

### 4.4 存储方案

| 方案 | 说明 | 成本 |
|------|------|------|
| **链上存储** | 只存哈希（32字节），不存文件 | 低 |
| **IPFS** | 去中心化存储，文件内容寻址 | 免费/低成本 |
| **Arweave** | 永久存储，一次付费 | ~$0.01/KB |
| **本地存储** | 用户自己保存文件 | 免费 |
| **云存储** | AWS S3 / 阿里 OSS | 按量付费 |

**推荐组合**：哈希上链 + 文件本地存储/IPFS + 元数据 JSON 本地保存

### 4.5 推荐架构：Local Chain + 公链锚定

```
┌─────────────────────────────────────────────┐
│                Evidence 系统                  │
├─────────────────────────────────────────────┤
│  用户上传文件 → SHA-256 → 写入本地链         │
│                          → 本地存储文件       │
│                          → 元数据 JSON        │
├─────────────────────────────────────────────┤
│  定期（每天/每周）：                          │
│  本地链 Merkle Root → 写入公链（L2）          │
├─────────────────────────────────────────────┤
│  验证时：                                    │
│  本地验证（快速） + 公链验证（可选）          │
└─────────────────────────────────────────────┘
```

## 五、功能设计建议

### 5.1 MVP 功能

1. **文件上传** — 支持图片、视频、音频、文字
2. **自动哈希** — 上传即计算 SHA-256
3. **存证记录** — 哈希 + 时间戳 + 文件元数据
4. **验证功能** — 上传文件，验证哈希是否匹配
5. **存证证书** — 生成可下载的存证证明（PDF/JSON）

### 5.2 进阶功能

6. **批量存证** — 一次上传多个文件
7. **公链锚定** — 定期将哈希写入公链
8. **分享验证** — 生成验证链接，他人可验证
9. **EXIF 提取** — 从照片提取拍摄时间/地点/GPS
10. **水印嵌入** — 在图片中嵌入隐式哈希水印

### 5.3 技术栈建议

| 层 | 技术 |
|------|------|
| 前端 | React/Next.js 或纯 HTML（MVP） |
| 后端 | Node.js / Go |
| 哈希 | Web Crypto API（前端）/ crypto 模块（后端） |
| 本地链 | LevelDB + 简单链结构 |
| 公链锚定 | Ethers.js（Ethereum L2） |
| 存储 | 本地文件系统 + IPFS（可选） |
| 数据库 | SQLite（本地）/ PostgreSQL（服务端） |

## 六、关于"本地链"的深入分析

### 优势

- 零成本，无 Gas
- 秒级确认
- 数据不出本地，隐私好
- 可以自定义链结构和共识

### 劣势

- **法律效力存疑**：单方控制的链，法院可能不认可
- **篡改风险**：如果只有一台机器，理论上可以改写
- **不可对外证明**：别人无法独立验证

### 解决方案

1. **多设备同步** — 至少 3 个节点互相验证
2. **公链锚定** — 定期把根哈希写入公链，提供外部验证
3. **数字签名** — 用私钥签名，证明是本人操作
4. **可信时间戳** — 从权威时间源获取时间
